# M7 任务板：验证规范与增量证据

## 目标

建立 Alpha 1.1 的验证闭环，确保难度、速度、AI 行为和流程体验的改进都能被复现、比较和追溯。M7 不改变游戏机制，它负责定义“怎样证明改进真的发生”。

## 当前状态

`Done`

已有 Alpha 1.1 聚合脚本 `scripts/alpha1.1_acceptance.py`，但当前覆盖范围仍偏基础，尚不足以证明速度优化、难度行为差异和多人局体验增量真实存在。

## 第一性原理

玩家感知到的改进来自实际行为，而不是实现意图。验证系统必须从玩家体验倒推：

- 速度改进必须表现为等待时间下降，而不是只减少某个函数的耗时。
- 难度改进必须表现为不同难度下 AI 决策质量、噪声和协作程度不同。
- 稳定性改进必须表现为长流程不重复弹窗、不丢状态、不破坏结算。
- 发布质量必须表现为同一组命令可以复现通过结果。

## 任务列表

### A11-VERIFY-029：建立 Alpha 1.1 验证规范

- 状态：`Done`
- 类型：文档 / 流程
- 范围：新增验证政策，定义证据等级、Done 条件和证据文件规范。
- 验收：
  - `docs/alpha-1.1-plan/verification_policy.md` 存在。
  - 主计划与任务板索引能链接到该规范。
  - 规范明确区分静态检查、行为验收和基准测试。

### A11-VERIFY-030：升级 Alpha 1.1 聚合验收入口

- 状态：`Done`
- 类型：脚本 / 回归保护
- 范围：逐步把新增验收脚本接入 `scripts/alpha1.1_acceptance.py`。
- 验收：
  - 聚合脚本输出每个验收项的名称、命令、耗时和结果。
  - 缺失的计划中脚本不能被误报为通过。
  - 失败时能清楚指出是规则、难度、速度还是文档证据问题。

### A11-VERIFY-031：补齐难度行为验收

- 状态：`Done`
- 类型：测试 / AI 行为
- 范围：新增或完善固定场景验收，比较不同难度下的 AI 决策差异。
- 推荐脚本：`scripts/difficulty_behavior_acceptance.py`
- 验收：
  - 至少覆盖新手、普通、老手、专家四档。
  - 至少比较发言质量、提名倾向、投票一致性或噪声触发率中的两项。
  - 输出可读摘要，说明差异是否符合预期。

### A11-VERIFY-032：补齐 AI 速度验收

- 状态：`Done`
- 类型：测试 / 性能
- 范围：验证多人局下 AI 发言、提名、投票的响应预算。
- 推荐脚本：`scripts/ai_speed_acceptance.py`
- 验收：
  - 至少覆盖 5 人局和 10 人局。
  - 输出 P50、P95、最大耗时和超时次数。
  - 能比较串行、并发或缓存策略的差异。
  - 结果可写入证据文件。

### A11-VERIFY-033：建立基线与证据模板

- 状态：`Done`
- 类型：文档 / 数据留存
- 范围：建立 `docs/alpha-1.1-evidence/` 证据目录和模板。
- 验收：
  - 每个完成项能记录任务编号、命令、结果、基线、残留风险。
  - 证据文件不依赖临时日志或本地机器绝对路径。
  - 证据目录不混入运行时数据库、session 或一次性调试数据。

### A11-VERIFY-034：补齐阵营信息边界测试

- 状态：`Done`
- 类型：测试 / Prompt 安全边界
- 范围：验证 Master、邪恶阵营、善良阵营在不同难度下不会获得不该知道的信息。
- 验收：
  - [x] 测试覆盖 prompt 构造、AI 决策输入和可见信息摘要。
  - [x] Master 伪装增强不能泄漏邪恶队友身份。
  - [x] 善良阵营不能因为难度提高而获得隐藏阵营真相。

### A11-VERIFY-035：建立 Alpha 1.1 发布证据索引

- 状态：`Done`
- 类型：文档 / Release
- 范围：在发布前汇总每个关键任务的证据文件。
- 验收：
  - 主计划能链接到发布证据索引。
  - 每个 P0/P1 项都有对应证据或明确豁免说明。
  - 发布说明不使用无法复现的主观描述。

### A11-VERIFY-036：建立人工试玩记录流程

- 状态：`Planned` → 延期至真人参与时执行
- 类型：人工验证 / 体验
- 范围：定义真实试玩时需要记录的内容，补足自动化难以覆盖的体验判断。
- 验收：
  - 记录对局人数、难度、AI 等待时间体感、结算流程、异常现象。
  - 将人工结论与自动化指标分开记录。
  - 不把单局主观反馈直接当成性能结论。

## 推荐推进顺序

1. 先完成 A11-VERIFY-029 与 A11-VERIFY-033，统一证据格式。
2. 再完成 A11-VERIFY-031 与 A11-VERIFY-032，把难度和速度从“计划项”变成可测行为。
3. 然后完成 A11-VERIFY-030，把成熟脚本纳入聚合验收。
4. 发布前完成 A11-VERIFY-035 与 A11-VERIFY-036。

## 完成标准

M7 完成时，Alpha 1.1 的主要改进项必须能回答：

- 改了什么。
- 为什么认为它变好了。
- 用什么命令复现。
- 和什么基线相比。
- 有没有破坏 Alpha 1.0 的规则与流程。
