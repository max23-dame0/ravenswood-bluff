# Alpha 1.1 验证规范：证明改进真实存在

本文档定义 Alpha 1.1 改进项的验证规则。任何任务从 `Planned` 或 `In Progress` 移动到 `Done` 前，都必须留下可复现证据，证明改进不只是字段、文案或配置存在，而是在行为、速度、难度或稳定性上产生了真实增量。

## 1. 验证目标

Alpha 1.1 的核心风险不是“没有代码”，而是“代码看起来生效，但玩家体验没有变好”。因此验证必须回答四个问题：

1. 改进前后的差异是否可观察。
2. 差异是否来自本次改动，而不是随机波动。
3. 差异是否没有破坏 Alpha 1.0 已有规则与流程。
4. 差异是否能被后续开发者用同一命令或同一场景复现。

## 2. 不可跳过的规则

- 不能把“配置字段存在”当成“难度生效”。
- 不能把“prompt 中出现关键词”当成“AI 行为改变”。
- 不能把“单次响应变快”当成“多人局整体变快”。
- 不能只保留主观结论，必须记录命令、场景、结果和证据文件。
- 不能只测成功路径，至少要覆盖一条回归路径，证明 Alpha 1.0 的核心流程没有被破坏。

## 3. 证据等级

| 等级 | 名称 | 说明 | 可用于 Done 判定 |
| --- | --- | --- | --- |
| L0 | 设计说明 | 计划、任务拆分、风险描述 | 否 |
| L1 | 静态契约 | schema、配置、prompt、文案、导出字段检查 | 只能辅助 |
| L2 | 单元测试 | 纯函数、策略选择、预算分配、边界条件测试 | 可用于低风险项 |
| L3 | 行为验收 | 固定局面、固定角色、固定难度下的可重复行为检查 | 必须用于 AI 行为项 |
| L4 | 基准与烟测 | 多人 mock 对局、延迟统计、回合推进、聚合验收脚本 | 必须用于速度与流程项 |
| L5 | 人工试玩记录 | 手动测试、真实局反馈、截图或日志摘要 | Release 前建议保留 |

## 4. 改进类型与最低证据要求

| 改进类型 | 最低证据 | 推荐命令或产物 |
| --- | --- | --- |
| 难度配置 | L2 + L3 | `pytest tests/test_difficulty.py tests/test_decision_noise.py`，`python scripts/difficulty_acceptance.py` |
| 难度行为差异 | L3 + 对照组 | `python scripts/difficulty_comparison.py`，计划补齐 `scripts/difficulty_behavior_acceptance.py` |
| AI 速度优化 | L4 + 基线对比 | `python scripts/parallel_benchmark.py`，计划补齐 `scripts/ai_speed_acceptance.py` |
| Master 伪装与阵营边界 | L2 + L3 | prompt dump 样本，阵营信息隔离测试 |
| UI 与结算流程 | L3 + 手动记录 | 浏览器流程验收，结算关闭后不重复弹出 |
| 发布聚合验收 | L4 | `python scripts/alpha1.1_acceptance.py` |

## 5. 增量真实存在的判定

一个优化项只有同时满足以下条件，才能标记为完成：

1. 有明确的基线：旧实现、默认配置、低难度或串行执行作为对照。
2. 有明确的观测指标：响应耗时、并发批次、行为选择率、噪声触发率、规则违规数等。
3. 有至少一次可复现命令：命令必须写入任务板或证据文件。
4. 有结果摘要：记录通过、失败、偏差或仍需人工判断的部分。
5. 有回归保护：证明 Alpha 1.0 的核心规则、结算、重放、记录或 live backend 没有被破坏。

## 6. 证据文件规范

后续证据统一放入：

```text
docs/alpha-1.1-evidence/
```

目录索引与模板见 [../alpha-1.1-evidence/README.md](../alpha-1.1-evidence/README.md)。

建议命名：

```text
YYYYMMDD_<task_id>_<short_slug>.md
```

证据文件应至少包含：

- 任务编号。
- 改动范围。
- 验证命令。
- 关键结果。
- 对照组或基线。
- 是否通过 Done 判定。
- 残留风险。

## 7. Done 检查清单

每个 Alpha 1.1 任务完成前，任务板中必须能追溯到：

- [ ] 对应代码或文档变更。
- [ ] 对应测试命令。
- [ ] 对应证据等级。
- [ ] 对应结果摘要。
- [ ] 对应残留风险或后续任务。

## 8. 聚合验收原则

`scripts/alpha1.1_acceptance.py` 是最终发布前的聚合入口。它应逐步纳入：

- Alpha 1.0 规则回归。
- 难度配置与行为验收。
- AI 速度基准与响应预算验收。
- 关键 UI/结算流程回归。
- 文档与任务板证据完整性检查。

在所有计划脚本补齐前，不能把聚合脚本通过视为 Alpha 1.1 全量通过，只能视为“当前已接入验收项通过”。
