### 4.2 问题

1. **Vote 和 Nomination 100% 走 fallback**: 所有 vote/nomination 的 LLM 调用都超时，走的是默认策略
2. **Speak 无 fallback**: 所有发言都成功调用 LLM，但延迟高
3. **Fallback 原因**: 全部是 `orchestrator_hard_timeout:vote` 和 `orchestrator_hard_timeout:nomination_intent`
4. **影响**: 投票行为完全由默认逻辑决定，AI 无法根据场上局势做出策略性投票

## 7. 未来优化计划

### 7.1 Token 优化 (预计节省 40-60%)

| 优化项 | 预期效果 | 优先级 |
|--------|---------|--------|
| **Prompt 压缩**: 对历史发言做摘要而非全文注入 | 减少 prompt token 30-50% | P0 |
| **讨论轮数动态调整**: 根据局势减少讨论轮数 | 减少总 speak 次数 | P1 |
| **死后发言限制**: 限制死亡玩家发言次数或长度 | 减少无效 token 消耗 | P1 |
| **Context window 管理**: 滑动窗口 + 关键信息保留 | 控制 prompt 增长 | P0 |
| **重复内容检测**: 跳过与前一轮高度相似的发言 | 减少冗余 token | P2 |

### 7.2 延迟优化 (目标: Speak P50 < 5s)

| 优化项 | 预期效果 | 优先级 |
|--------|---------|--------|
| **流式响应**: 使用 SSE/WebSocket 流式输出发言 | 感知延迟降低 60-80% | P0 |
| **并行预生成**: 在等待其他人发言时预生成自己的发言 | 减少实际等待时间 | P1 |
| **Vote/Nomination LLM 超时调优**: 增加超时或使用更快模型 | 减少 fallback 率 | P0 |
| **模型选择优化**: 简单决策用小模型，复杂发言用大模型 | 降低延迟和成本 | P1 |
| **Prompt 缓存**: 利用 Anthropic/OpenAI 的 prompt caching | 减少重复计算 | P0 |

### 7.3 AI 质量优化

| 优化项 | 预期效果 | 优先级 |
|--------|---------|--------|
| **从众型人设增强**: 僧侣等从众角色需有底线逻辑 | 避免被邪恶方完全利用 | P1 |
| **信息整合提示**: 在 prompt 中引导 AI 综合多条信息链 | 提升推理质量 | P1 |
| **讨论疲劳检测**: 检测重复讨论并提前结束 | 提升讨论效率 | P2 |
| **投票策略注入**: vote fallback 应考虑场上局势 | 提升投票质量 | P0 |
| **死后影响力衰减**: 死人发言权重降低 | 减少死后干扰 | P2 |

### 7.4 架构优化

| 优化项 | 预期效果 | 优先级 |
|--------|---------|--------|
| **多模型路由**: speak 用大模型，vote 用小模型 | 平衡质量和速度 | P1 |
| **异步 LLM 调用池**: 并行处理多个 agent 的 LLM 请求 | 减少串行等待 | P1 |
| **Token 预算系统**: 每轮设定 token 上限，超出则简化 prompt | 控制成本 | P2 |
| **A/B 测试框架**: 自动对比不同 prompt 策略的效果 | 持续优化 | P2 |
